Resumen del proyecto

Bandyou es una red social y directorio para la comunidad musical española. Conecta músicos independientes, bandas, salas de conciertos, profesores de música y locales de ensayo en un solo lugar. El objetivo es que cualquier músico pueda encontrar lo que necesita — compañeros de banda, salas donde tocar, clases, o vender su equipo — sin salir de la plataforma.

Es un proyecto personal (SPA sin backend propio) que utiliza Supabase como capa de datos y autenticación, con Angular 18 en el frontend. Está diseñado para poder escalar a producción real con usuarios sin necesidad de infraestructura propia.

Arquitectura y estructura

El proyecto sigue una arquitectura Angular moderna con standalone components (sin NgModules), lazy loading en cada ruta, y gestión de estado local mediante Angular Signals. No hay estado global centralizado (sin NgRx), lo que simplifica el código a costa de algo de coordinación entre componentes.

│   ├── models/         # Interfaces TypeScript de todos los modelos

Patrón de routing
Todas las rutas usan loadComponent() con imports dinámicos. Las rutas protegidas tienen canActivate: [authGuard]. No existe módulo de routing separado — todo está en app.routes.ts.
05

Servicios core

Todos los servicios son providedIn: 'root' (singletons). Ninguno usa stores externos — el estado vive en signals o se recarga on-demand.

SupabaseService
Singleton que wrappea el cliente Supabase. Todas las demás clases usan this.supabase.client para acceder a la BD y this.supabase.auth para autenticación.
AuthService
Gestiona la sesión con signals: session, user, isLoggedIn. Escucha cambios de estado con onAuthStateChange. Métodos: login, register, Google OAuth, logout.
MessagesService
CRUD completo de conversaciones y mensajes. Convenciones de clave: user1_id < user2_id (lexicográfico). Incluye subscripción Realtime, marcado como leído, y creación automática de notificación al enviar.
NotificationsService
Tabla notifications + canal Realtime. Expone unreadCount como signal reactivo. Tipos soportados: message, application, rsvp, review, system.
FavoritesService
Toggle de favoritos para cualquier entidad (musician, band, venue, teacher, rehearsal). Constraint único (user_id, entity_type, entity_id).
ToastService
Notificaciones de UI (éxito / error / info). Componente ToastComponent renderiza la cola de toasts en la esquina de la pantalla.
SeoService
Gestiona <title> y meta tags (description, OG) para cada ruta. Se llama en el ngOnInit de cada componente que lo necesite.

Flujo de autenticación

✓
Registro con email/password
Supabase Auth. Tras registrarse, el usuario es redirigido a /onboarding para crear su perfil.
✓
Login con email/password
Redirige a /home tras autenticarse correctamente.
✓
Login con Google (OAuth)
Redirect a /home. Configurado en SupabaseService con redirectTo: origin/home.
✓
Recuperación de contraseña
Flujo completo: forgot-password → email → reset-password con token.
✓
Guard de rutas protegidas
authGuard redirige a /auth/login si el usuario no está autenticado. Rutas protegidas: home, dashboard, inbox, chat, favorites, notifications, events/create, shop/new, onboarding.
✓
Onboarding de perfil
El rol se guarda en localStorage antes de registrarse. OnboardingComponent detecta si el usuario ya tiene perfil (isEditing) y permite editarlo o crearlo por primera vez.

Funcionalidad en tiempo real

Supabase Realtime se usa mediante canales de postgres_changes. Se suscriben automáticamente al entrar a la pantalla y se destruyen al salir.

FuncionalidadCanalTabla escuchada
Chat en tiempo real messages:{conversationId} messages — INSERT filtrado por conversación
Notificaciones en vivo notifications:{userId} notifications — INSERT filtrado por destinatario
Inbox (nuevos mensajes) inbox-updates-{random} messages — INSERT global, filtra por remitente
Punto a mejorar
El canal de inbox usa un nombre aleatorio (Math.random()), lo que puede crear múltiples canales huérfanos si el componente se re-renderiza. Sería mejor un ID estable basado en el userId.

Políticas de seguridad (RLS)

Patrón general aplicado en todas las tablas:

OperaciónPolítica
SELECTPúblico (USING (true)) para perfiles, eventos, posts, gear. Privado (user_id = auth.uid()) para favoritos, notificaciones, mensajes.
INSERTRequiere autenticación. Se valida user_id = auth.uid().
UPDATE/DELETESolo el propietario del recurso. En tablas relacionadas (e.g. band_members), se valida via JOIN que el usuario sea dueño de la banda.
Punto a revisar
La política de DELETE en conversations puede no estar configurada correctamente — hay un warning en MessagesService.deleteConversation() que detecta cuando se eliminan 0 filas. Revisar las políticas RLS de la tabla conversations en Supabase.

Gaps y puntos de mejora detectados

⚠
Cobertura de tests muy baja (~15%)
Solo existen tests unitarios para los servicios core. No hay tests de componentes ni E2E. Riesgo real en producción.
⚠
Bug RLS en delete de conversaciones
La tabla conversations puede tener una política de DELETE incompleta. Hay un warning en deleteConversation() que lo indica.
!
Canal Realtime de inbox con nombre aleatorio
Puede crear canales huérfanos si el componente se destruye y recrea. Solución: usar userId como ID del canal.
!
Búsqueda textual del lado cliente
El filtro por query en fetchPage() se aplica en JS sobre los resultados ya paginados. Si el resultado que coincide está en la página 3, no aparece. Solución: usar ilike en la query de Supabase.
!
Sin validación de imágenes en la tienda
El campo images acepta URLs de texto libre sin validar. No hay upload de imágenes real en la tienda (solo en avatar del perfil).
!
Uso de any en varios componentes
El dashboard y otros componentes usan any para datos de Supabase en lugar de los tipos de core/models/index.ts.
i
Plan gratuito de Supabase
La base de datos se pausa a los 7 días de inactividad. Para producción real, migrar al plan Pro (~$25/mes).
i
Sin sistema de moderación de contenido
No hay forma de reportar perfiles, posts o mensajes inapropiados. Necesario antes de abrir al público.
i
Sin rate limiting en cliente
Supabase provee rate limiting en su capa, pero no hay throttling explícito en acciones como envío de mensajes o creación de posts.
